1. DDS Overview

The Dynamic Decision System ("DDS") is an enterprise-grade identity and multi-factor authentication (MFA) platform designed for the modern web.

Similar to tools like Duo and Okta Verify, DDS shifts the authentication paradigm away from friction-heavy OTP text messages and hardware tokens, replacing them with seamless, secure, push-notification-based approvals routed directly to the DDS Mobile App.

2. Mission & Vision

Our Mission

To empower developers with plug-and-play, zero-configuration identity controls that protect user applications from credential harvesting and SIM-swapping attacks.

Our Vision

To make passwordless verification the global standard, ensuring every digital interaction is secured by strong, asymmetric, public-key cryptography with zero friction for the end-user.

3. Platform Architecture

DDS utilizes a distributed, real-time microservices architecture designed to process verification sessions with sub-second latency.

  • Developer Website/Backend: Interacts with DDS via the official SDK packages (e.g., @dds/server).
  • DDS Server: Central authentication engine handling requests verification, database operations, and real-time WebSocket messaging.
  • Real-time Broker: Manages state-synchronized connections with both the Developer Portal and the DDS Mobile App for instant push notifications.
  • DDS Mobile Application: The user-facing app that processes cryptographic approvals.

4. Authentication Model

The core DDS authentication flow operates as follows:

  1. A developer website requests authentication for a specific user using their mobile number.
  2. The DDS Server validates the developer's credentials (App ID, Public Key, Secret Key) and returns a verification code.
  3. The developer website immediately displays this code.
  4. The user receives a secure push prompt on their verified DDS Mobile App.
  5. The user inputs the matching verification code and taps Approve.
  6. DDS Server verifies the signature, updates the status, and the developer website completes the login.

5. Security Principles

DDS is built on a foundation of strict security guidelines:

  • Asymmetric Key Pairs: Verification payloads are signed and confirmed cryptographically on-device.
  • Credential Isolation: Developers must keep Secret Keys inside secure backend environments. The DDS SDK strictly rejects client-side initialization to prevent token leaks.
  • Hashed Credentials: API credentials are verified against one-way SHA-256 signatures, ensuring even in the case of system intrusions, secrets remain uncompromised.

6. Why DDS?

DDS provides enterprise features that streamline operations and offer unmatched developer experiences:

  • Pay As You Go Scaling: Avoid high fixed costs. Pay only for successful authentication requests.
  • Zero-Config SDK: Initialized with a single CLI command (npx dds init), configuring routes and clients instantly.
  • Grace Period Protections: 7-day payment window prevents service interruptions when card authorizations fail.
